Automated REST API testing has traditionally focused on functional compliance, whereas security assurance has received comparatively limited attention. Existing approaches to API security testing largely operate at the syntactic level of HTTP requests and responses, limiting their ability to identify vulnerabilities whose manifestation depends on the semantic behavior of the target system. This limitation is especially significant given that four of the top five categories in the OWASP API Security Top 10 (2023) encompass weaknesses that cannot be reliably detected through syntactic analysis alone. Their identification requires evaluation criteria grounded in both the semantics of the security controls under assessment and domain-specific knowledge. Despite its importance, the systematic definition of such criteria remains one of the least explored challenges in the literature. Existing solutions further exhibit a recurring trade-off between portability and semantic depth, while their lack of deterministic execution undermines reproducibility and limits integration into continuous development and verification workflows.

To address these limitations, this work proposes a contract-driven approach to REST API security assurance. The approach leverages the target system's OpenAPI specification to generate syntactically valid security probes and derives evaluation criteria directly from the semantics of the security controls being assessed. These criteria are embedded within the test logic, enabling automated verification of security properties beyond protocol-level compliance. At runtime, the framework relies exclusively on two sources of target knowledge, the OpenAPI specification and a configuration file, allowing it to operate on any documented REST API without source-code access or target-specific customizations. Moreover, fixed oracles and deterministic execution by design ensure that assessment results are reproducible, operator-independent, and consistent across repeated executions. Security checks are organized according to an eight-domain taxonomy that structures the assessment process and automatically scales coverage according to the privileges available during testing.

The proposed approach was validated on Forgejo 14.0.3 deployed behind Kong 3.9, a realistic target exposing a native OpenAPI specification. The evaluation was conducted on a dedicated testbed designed to produce both expected successes and failures across the assessed security controls. Results show that the framework correctly classified all test cases and additionally revealed a discrepancy between the requirements declared in the OpenAPI specification and the actual behavior of the target system. Independent executions produced identical outputs, empirically confirming the deterministic and reproducible nature of the assessment process.

The contribution of this work is threefold. First, it introduces an implementation-agnostic eight-domain taxonomy for REST API security assurance. Second, it provides a systematic methodology for defining specified oracles in API security testing, deriving evaluation criteria explicitly from security semantics and domain knowledge. Third, it presents a deterministic DAG-based assessment engine that guarantees semantically correct execution ordering while preserving reproducibility. The proposed framework transforms security assurance into a reproducible and automatable activity that can be integrated directly into modern software delivery pipelines through machine-actionable assessment outcomes and semantic exit codes.
