% Context and Gaps
The vast majority of automated REST API testing addresses functional conformance rather than security; the few tools targeting security testing operate on the syntax of HTTP requests and fail to detect semantic vulnerabilities, which are opaque to syntactic analysis and detectable only through knowledge of the expected system behavior. Four of the first five risk categories in the OWASP API Security Top 10 2023 belong to this class. Detecting them automatically requires evaluation criteria grounded in the domain knowledge of each specific security control: building such criteria systematically is the most consistently overlooked challenge in the literature. Compounding these limitations are the recurring trade-off between portability and semantic depth in specialized tools, and the absence of deterministic reproducibility, which precludes integrating verification into continuous development cycles.

% Methodological Approach
In response, a contract-driven approach to REST API security assurance is proposed. The target's OpenAPI specification guides the construction of semantically significant probes; the evaluation criterion for each test is derived from the semantics of the specific security check being assessed and embedded in the logic of the corresponding test. The only two sources of target knowledge at runtime are the OpenAPI specification and the configuration file, making the system applicable to any documented REST API without code modifications. Fixed oracles and deterministic behavior by construction ensure the assessment output is operator-independent and reproducible across successive executions. Checks are organized into an eight-domain security taxonomy, with coverage depth scaling automatically to the access privileges available.

% Results and Contributions
Validation on Forgejo~14.0.3 exposed through Kong~3.9, a real target with a native OpenAPI specification, was conducted on a testbed designed to yield both intended passing and failing outcomes; the system correctly classified both cases, also detecting a real discrepancy between the specification's stated requirements and actual target behavior. Two independent runs confirmed the reproducibility of the output. Among the main contributions are an eight-domain security assurance taxonomy applicable regardless of the underlying implementation, a systematic approach to \emph{specified oracles} in REST API security testing with criteria derived from domain knowledge, and a deterministic DAG-based assessment engine ensuring semantically correct execution ordering. The empirically verified properties of determinism and reproducibility make security assurance actionable within the development cycle through semantic exit codes.
